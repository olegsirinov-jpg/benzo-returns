# Система обміну та повернення товарів

Окремий веб-сервіс оформлення, обліку та контролю повернень для інтернет-магазину на Хорошоп.
PHP 7.4 + MySQL, без фреймворку та composer-залежностей.

---

## Швидкий старт (OSPanel)

1. Створіть базу `povirnenya` (utf8mb4):
   ```sql
   CREATE DATABASE povirnenya CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
   ```
2. Перевірте `.env` — там уже налаштовано `DB_USER=root`, порожній пароль, `APP_URL=http://povirnenya`.
3. Відкрийте <http://povirnenya/install.php>, створіть адміністратора.
4. **Видаліть `install.php`.**
5. Увійдіть: <http://povirnenya/admin/login>

Публічна частина: <http://povirnenya/returns>

## Налаштування інтеграцій (.env)

### SalesDrive

```
SD_ENABLED=1
SD_URL=https://вашдомен.salesdrive.me
SD_API_KEY=...      # Установки → Загальні налаштування і інтеграції → Інші сервіси → API → API-ключі
SD_FORM_KEY=...     # ключ бази заявок, потрібен щоб додавати коментарі
```

Права ключа: **читання заявок** (пошук замовлення) + **оновлення заявок** (коментарі).

**Ліміт API SalesDrive на список заявок: 10 запитів/хв, 100/год, 1000/добу.**
Тому відповіді кешуються на 15 хв (таблиця `sd_cache`), а виклики рахуються в `sd_calls`
з внутрішнім лімітом 8/хв. Якщо ліміт вичерпано — заявка все одно створиться,
але позначиться як «Потребує ручної перевірки». Клієнт ніколи не бачить помилку.

> **Важливо про коментарі:** `api/order/update/` **перезаписує** поле `comment`.
> Система спершу читає поточний коментар і дописує свій текст у кінець,
> щоб не затерти нотатки менеджерів.

Якщо API виявиться нестабільним — альтернатива з ТЗ (п.14.1): приймати вебхуки SalesDrive
у локальну базу і шукати замовлення там. Точки розширення в `app/SalesDrive.php::findOrder()`.

### Telegram

1. Створіть бота через [@BotFather](https://t.me/BotFather) → отримаєте токен.
2. Додайте бота в групу менеджерів.
3. Дізнайтесь `chat_id`: `https://api.telegram.org/bot<TOKEN>/getUpdates` (для груп — від'ємне число).

```
TG_ENABLED=1
TG_BOT_TOKEN=123456:AA...
TG_CHAT_ID=-1001234567890
```

Сповіщення: нова заявка, ТТН від клієнта, нагадування про зависання >24 год.

### Крон — нагадування про завислі заявки

Щогодини — нагадування про зависання:
```
0 * * * * php /шлях/до/проєкту/cron/stale.php
```

Кожні 30 хв — трекінг зворотних посилок Нової пошти (авто-статуси in_transit / received):
```
*/30 * * * * php /шлях/до/проєкту/cron/nptrack.php
```

Або по HTTP (тоді приберіть `cron` з правила блокування в `.htaccess`):
```
https://returns.benzo-pila.in.ua/cron/stale.php?key=<CRON_KEY>
```

## Деплой на хостинг

1. Залийте всі файли в корінь субдомену (`returns.benzo-pila.in.ua`).
2. Створіть базу, оновіть `.env` (`APP_URL=https://returns.benzo-pila.in.ua`, `APP_DEBUG=0`).
3. Запустіть `install.php`, потім **видаліть його**.
4. Права: `uploads/` і `storage/logs/` — на запис (755/775).
5. Якщо PHP працює через **FPM/CGI** — директиви `php_value` в `.htaccess` не спрацюють
   (вони обгорнуті в `IfModule`, помилки не буде). Пропишіть у `php.ini` / `.user.ini`:
   ```ini
   upload_max_filesize = 12M
   post_max_size = 60M
   max_file_uploads = 25
   ```

## Логіка статусів (п.10.1 ТЗ)

```
new → (менеджер відкрив картку) → manager_review
    → need_more_info   (потрібні фото)
    → approved         → waiting_customer_shipment
                       → (клієнт вніс ТТН) → in_transit
                       → received → inspection
                       → refund_pending → refunded → closed
    → rejected         (обов'язково: причина + коментар для клієнта)
```

Статус `manager_review` ставиться автоматично при першому відкритті картки.
`in_transit` — автоматично, коли клієнт вносить ТТН на сторінці статусу.

## Безпека

- CSRF-токени на всіх POST-формах.
- Захист від перебору: 5 спроб входу → блок на 10 хв; ліміти на пошук замовлення та створення заявок.
- Паролі — `password_hash` (bcrypt), сесійні cookie `HttpOnly` + `SameSite=Lax`.
- Усі SQL-запити — підготовлені вирази.
- Завантажені фото: перевірка через `getimagesize`, випадкові імена, заборона виконання скриптів.
- Службові каталоги (`app/`, `views/`, `config/`, `storage/`, `cron/`) і `.env` закриті в `.htaccess`.

**Персональні дані.** У базі зберігаються ПІБ, телефони, IBAN, ІПН. Це вимагає:
захищеного доступу до адмінки (краще ще й HTTPS + обмеження по IP),
регулярних бекапів і політики видалення старих заявок.

## Що не входить у цей етап

- Автоматичні SMS/Viber/email клієнту (у `Telegram.php` є готовий патерн для нового каналу;
  шаблони повідомлень з п.15.1 ТЗ уже використані на сторінці статусу).
- Експорт у `.xlsx` (є CSV з BOM — коректно відкривається в Excel).
- Інтеграція з API Нової пошти (поле ТТН заповнюється вручну / клієнтом).
- Синхронізація замовлень через вебхук SalesDrive.
